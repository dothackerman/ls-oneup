# Crypto Inventory Evidence Split

## Goal

Separate live crypto evidence from provenance and forbid ephemeral migration files from live evidence.

The PR 15 comment is a symptom, not the design target:

- the gate currently treats current evidence and provenance as the same field
- migration filenames are ephemeral and get renamed, squashed, or deleted
- a stale migration breadcrumb can fail `npm run crypto:run` even when the live crypto control is still correct

## Required Contract

The inventory must model three distinct evidence types.

### Generated live evidence

- `discovery`

This is generated by `npm run crypto:inventory`.
It is the primary machine-validated proof that the live crypto call sites still exist.

### Current supporting evidence

- `current_code_references`

These are stable, current references that explain the live control.
They must exist on disk and must be gate-validated.

Rule:

- `current_code_references` must not contain `migrations/*.sql`

### Provenance

- `provenance_code_references`

These are audit breadcrumbs such as migration lineage, renamed files, and deleted paths that explain how the control evolved.
They may point at deleted or renamed files.
They must not fail the gate.

### Schema version

- add `schema_version: 2` at the top level
- reject legacy `code_references` in schema v2 so the split stays explicit

## Non-Goals

- Do not solve this by broadening the gate to accept stale current refs.
- Do not make migration files stable by policy fiction; they are provenance, not live evidence.
- Do not keep patching deleted migration names one by one after each rename.
- Do not use README prose as the source of truth.

## Implementation Slices

### Slice 1: Gate semantics

Owned files:

- `scripts/crypto-gate.mjs`

Supporting files:

- `tests/security.unit.test.ts`
- `docs/security/crypto-policy.md`

Forbidden files:

- application runtime code
- `docs/security/asvs/**`

Work:

- require `schema_version: 2`
- validate `current_code_references` and `discovery`
- ignore `provenance_code_references` for path existence checks
- reject `migrations/*.sql` in `current_code_references`
- reject legacy `code_references` in schema v2
- add unit coverage for the new gate semantics

Done when:

- the gate fails if a live supporting reference is missing
- the gate fails if a migration file appears in live evidence
- the gate ignores deleted provenance references
- the gate only accepts the explicit schema-v2 field split

Done when:

- no entry keeps `migrations/*.sql` in `current_code_references`

## Acceptance Criteria

- `schema_version: 2` is present in the crypto inventory
- live evidence and provenance are separate fields
- `crypto:gate` validates only live evidence
- `crypto:gate` rejects `migrations/*.sql` inside live evidence
- the submission-encryption inventory entry no longer fails because of a deleted migration breadcrumb
- the security docs describe the green state instead of memorializing the old failure

## Risks

- If `current_code_references` becomes a junk drawer, the schema changes but the drift comes back.
- If provenance is omitted entirely, future reviewers lose the migration lineage that explains why a control exists.
- If ASVS docs continue to describe the old red state after the gate turns green, the repo will contradict itself and reviewers will stop trusting the evidence.
